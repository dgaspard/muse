## ğŸ”„ Versioning Policy

All specifications in `data-models/` follow semantic versioning:

- **schemas/** - Schema version increments require:
  - New version file (e.g., v1.1.md â†’ v1.2.md)
  - Migration notes for breaking changes
  - CODEOWNERS approval

- **normalization/** - Normalization rule changes require:
  - Version bump with migration path
  - Backward compatibility notes

- **errors/** - Error model changes require:
  - Version increment
  - Impact analysis on client applications

### Making Specification Changes

When updating specifications:

1. **Never edit existing version files in place** - Create new version file
2. **Label PRs** - Use `governance-change` label for spec changes
3. **Include migration notes** - Document upgrade path in PR description
4. **Get approval** - Spec changes require CODEOWNERS review
5. **Update metadata** - All artifacts must reference current spec versions

See [CI-CD Governance Enforcement](ci-cd/governance-enforcement.md) for detailed requirements.
